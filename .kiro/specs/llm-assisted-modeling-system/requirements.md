# LLM辅助建模系统需求文档

## 引言

本文档定义了基于大语言模型(LLM)的等离子体器件仿真软件辅助建模系统的需求。该系统旨在通过引入LLM技术，解决传统规则转译器在复杂建模语义处理中的规则爆炸问题，实现从MAGIC建模脚本到UNIPIC配置文件的智能转换。

## 术语表

- **LLM**: Large Language Model，大语言模型
- **MAGIC**: 等离子体器件仿真软件，使用MCL脚本语言
- **UNIPIC**: 等离子体器件仿真软件，使用.in配置文件
- **MCL**: MAGIC Command Language，MAGIC的脚本语言
- **中间语义文本**: 建模完成时的语义中介表示，连接MAGIC和UNIPIC的桥梁
- **ParseResult**: 解析器统一返回的结构化对象
- **规则爆炸**: 传统转译器中规则数量随器件类型和参数组合呈指数增长的问题
- **语义泛化**: LLM在有限规则基础上对未覆盖组合进行合理推断的能力

## 需求

### 需求 1

**用户故事:** 作为仿真工程师，我希望系统能够智能解析复杂的MAGIC命令，以便快速扩展对新命令类型的支持而无需编写复杂的语法规则。

#### 验收标准

1. WHEN 系统遇到未在PLY语法中定义的MAGIC命令 THEN 系统SHALL自动路由到LLM解析器进行处理
2. WHEN LLM解析器处理MAGIC命令 THEN 系统SHALL返回与PLY解析器相同格式的ParseResult结构
3. WHEN LLM解析器处理批量命令（5-8条同类型命令）THEN 系统SHALL支持并发API调用以提高处理效率
4. WHEN LLM解析失败或返回格式错误 THEN 系统SHALL提供清晰的错误信息并支持回退到其他解析器
5. WHEN 新的器件类型引入新命令 THEN 系统SHALL通过更新prompt模板快速支持新命令解析

### 需求 2

**用户故事:** 作为系统开发者，我希望建立统一的中间语义文本格式，以便实现MAGIC和UNIPIC之间的语义对齐和转换。

#### 验收标准

1. WHEN 系统处理MAGIC建模脚本 THEN 系统SHALL生成包含所有建模语义的中间文本表示
2. WHEN 中间语义文本包含变量定义 THEN 系统SHALL保存变量名、表达式、数值和单位信息
3. WHEN 中间语义文本包含几何实体 THEN 系统SHALL支持POINT、LINE、AREA、SELECTION等几何类型的完整描述
4. WHEN 中间语义文本包含物理参数 THEN 系统SHALL正确处理材料定义、网格参数、边界条件、诊断设置等
5. WHEN 系统生成UNIPIC配置文件 THEN 系统SHALL基于中间语义文本生成所有必需的.in文件

### 需求 3

**用户故事:** 作为仿真工程师，我希望系统能够处理复杂的语义依赖关系，以便正确转换相互关联的建模元素。

#### 验收标准

1. WHEN 系统处理具有依赖关系的建模元素 THEN 系统SHALL自动识别并包含所有依赖的前置元素
2. WHEN LLM转换器处理单个语义元素 THEN 系统SHALL构建包含上下文信息的完整语义块
3. WHEN 系统查找映射规则 THEN 系统SHALL支持通过语义检索、手动配置和LLM推荐等多种方式获取映射规则
4. WHEN 多个语义元素需要并发处理 THEN 系统SHALL确保不同元素的转换过程可以并行执行
5. WHEN 转换过程中出现规则冲突 THEN 系统SHALL使用LLM进行智能决策和规则选择

### 需求 4

**用户故事:** 作为系统用户，我希望系统具备高性能和可靠性，以便在生产环境中稳定运行。

#### 验收标准

1. WHEN 系统处理大型建模文件（>1000行命令）THEN 系统SHALL在合理时间内完成处理（<5分钟）
2. WHEN 系统进行LLM API调用 THEN 系统SHALL实现智能缓存机制避免重复调用
3. WHEN LLM API调用失败或超时 THEN 系统SHALL提供重试机制和降级策略
4. WHEN 系统处理多种器件类型 THEN 系统SHALL支持BWO、MILO、Vir、Mitl、TM03_60GHz、XCB_PB等至少6种器件
5. WHEN 系统运行过程中出现错误 THEN 系统SHALL提供详细的错误日志和调试信息

### 需求 5

**用户故事:** 作为研究人员，我希望系统能够验证LLM方法的有效性，以便证明其在转译场景中的价值。

#### 验收标准

1. WHEN 系统使用LLM方法处理复杂语义 THEN 系统SHALL达到85%以上的转换准确率
2. WHEN 系统处理新器件类型 THEN LLM方法SHALL展现出比纯规则方法更好的泛化能力
3. WHEN 系统面临规则爆炸问题 THEN LLM方法SHALL显著减少所需的显式规则数量
4. WHEN 系统扩展支持新命令类型 THEN LLM方法SHALL比传统方法减少50%以上的开发时间
5. WHEN 系统进行性能对比测试 THEN 系统SHALL提供详细的准确率、效率和成本分析报告

### 需求 6

**用户故事:** 作为系统维护者，我希望系统具备良好的可扩展性和可维护性，以便支持长期发展和功能扩展。

#### 验收标准

1. WHEN 需要添加新的MAGIC命令支持 THEN 系统SHALL通过配置prompt模板实现快速扩展
2. WHEN 需要修改转换规则 THEN 系统SHALL支持通过配置文件或数据库更新规则而无需修改代码
3. WHEN 系统需要集成新的LLM模型 THEN 系统SHALL提供统一的LLM接口支持模型切换
4. WHEN 系统需要支持新的目标格式 THEN 系统SHALL通过扩展中间语义文本格式实现支持
5. WHEN 系统需要部署到不同环境 THEN 系统SHALL提供容器化部署和配置管理支持
